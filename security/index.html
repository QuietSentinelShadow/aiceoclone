1 Threat Model

Running an autonomous AI agent introduces unique attack surfaces. Our threat model considers four primary categories that must be addressed at every layer of the stack.

Unauthorized Access

Attackers gaining access to the NullClaw gateway API, management dashboard, or underlying host. Mitigated through JWT authentication, network isolation, and Traefik ingress controls.

Container Escape

An AI agent or exploited process breaking out of the container sandbox to reach the host. Mitigated through non-root users, seccomp profiles, capability dropping, and read-only filesystems.

API Key Theft

Exfiltration of AI provider API keys, which could result in financial abuse. Mitigated through environment variable injection, AES-256-GCM encryption at rest, and never storing keys in config files or images.

Resource Abuse

Runaway AI generation consuming excessive CPU, memory, disk, or API tokens. Mitigated through Docker resource limits, NullClaw resource caps, and per-user quotas in the SaaS platform.

2 Container Security

Every container runs with the principle of least privilege. The compose file enforces non-root execution, read-only filesystems, dropped capabilities, and custom seccomp profiles.

Non-root Users

Both containers create dedicated service users at build time. The Dockerfile uses a numeric UID/GID (1000) so the container never needs root privileges.

Dockerfile.nullclaw 
# Create non-root user
RUN addgroup -g 1000 nullclaw && \
    adduser -u 1000 -G nullclaw -D nullclaw

# Switch to non-root before CMD
USER nullclaw

Read-only Filesystem

The NullClaw container runs with a read-only root filesystem. Only a small tmpfs is mounted for scratch operations, limited to 64 MB.

docker-compose.yml (nullclaw service) 
read_only: true
tmpfs:
  - /tmp/nullclaw:size=64M

Capability Dropping & Privilege Escalation Prevention

The no-new-privileges flag prevents any process inside the container from gaining additional privileges through setuid binaries or other escalation vectors.

docker-compose.yml (both services) 
security_opt:
  - no-new-privileges:true

deploy:
  resources:
    limits:
      memory: 256M
      cpus: "0.5"

Seccomp Profiles

A custom seccomp profile whitelists only the syscalls NullClaw actually needs. The default action is SCMP_ACT_ERRNO, which blocks all unlisted syscalls and returns an error instead of killing the process.

docker/security/seccomp-nullclaw.json (excerpt) 
{
  "defaultAction": "SCMP_ACT_ERRNO",
  "architectures": ["SCMP_ARCH_X86_64", "SCMP_ARCH_AARCH64"],
  "syscalls": [{
    "names": [
      "read", "write", "open", "close",
      "socket", "connect", "bind", "listen",
      "epoll_create", "epoll_ctl", "epoll_wait",
      /* ...~60 total allowed syscalls... */
    ],
    "action": "SCMP_ACT_ALLOW"
  }]
}

To apply the profile, add to the NullClaw service:

security_opt:
  - no-new-privileges:true
  - seccomp=security/seccomp-nullclaw.json

3 Network Security

Network-level isolation is the first line of defense. Containers communicate over an internal bridge network, and all external traffic flows through Traefik as a reverse proxy.

Isolated Docker Networks

Each deployment uses a dedicated bridge network. Containers on different networks cannot communicate. In the SaaS platform, each tenant instance gets its own isolated network.

networks:
  content-net:
    driver: bridge
    internal: false  # set to true to block all egress

Traefik as Sole Ingress

In production, only Traefik exposes ports to the host. All other services are accessible only through Traefik routes, which provide TLS termination, request routing, and middleware chains.

traefik:
  image: traefik:v3.3
  ports:
    - "443:443"   # HTTPS only in production
  command:
    - "--providers.docker=true"
    - "--providers.docker.exposedbydefault=false"
    - "--entrypoints.websecure.address=:443"

Rate Limiting & CORS

Traefik middleware enforces per-IP rate limits to prevent brute-force and API abuse. CORS headers restrict which origins can call the API.

labels:
  # Rate limit: 100 requests per second, burst of 200
  - "traefik.http.middlewares.ratelimit.ratelimit.average=100"
  - "traefik.http.middlewares.ratelimit.ratelimit.burst=200"

  # CORS: restrict to dashboard origin
  - "traefik.http.middlewares.cors.headers.accesscontrolalloworiginlist=https://dashboard.example.com"
  - "traefik.http.middlewares.cors.headers.accesscontrolallowmethods=GET,POST,DELETE"

4 Application Security

The SaaS platform API implements defense-in-depth at the application layer with industry-standard libraries and cryptographic primitives.

JWT Authentication

Stateless tokens with 24-hour expiry using the jsonwebtoken library. Tokens contain only the user ID and email, no sensitive data.

jwt.sign(
  { userId, email },
  JWT_SECRET,
  { expiresIn: "24h" }
)

Bcrypt Password Hashing

Passwords are hashed with bcrypt using a cost factor of 12. Even if the database is compromised, passwords cannot be reversed.

const hash = await bcrypt.hash(
  password, 12
)

AES-256-GCM Encryption

API keys stored in the database are encrypted at rest with AES-256-GCM. The encryption key is injected via environment variable and never stored in the database or config files.

encrypt(apiKey, ENCRYPTION_KEY)
// iv:authTag:ciphertext

Zod Validation

All API inputs are validated with Zod schemas before processing. Invalid payloads are rejected with structured error messages before they reach any business logic.

const schema = z.object({
  email: z.string().email(),
  password: z.string().min(8),
})

Helmet.js HTTP Headers

The Express API uses Helmet.js to set secure HTTP headers. This protects against XSS, clickjacking, MIME sniffing, and other common web vulnerabilities.

X-Content-Type-Options
X-Frame-Options
Strict-Transport-Security
Content-Security-Policy
X-XSS-Protection
Referrer-Policy

5 NullClaw-Specific Security

NullClaw has its own multi-layer sandbox system that operates independently of Docker isolation. This provides defense-in-depth: even if Docker is misconfigured, NullClaw enforces its own restrictions.

Built-in Sandbox (Landlock / Bubblewrap / Docker)

NullClaw auto-detects the best available sandbox backend. On modern Linux kernels, it uses Landlock LSM for filesystem access control. On older kernels, it falls back to Bubblewrap (bwrap). Inside Docker, it leverages the existing container isolation.

Landlock
Kernel 5.13+ LSM
Bubblewrap
User-namespace sandbox
Docker
Container-native isolation

Workspace Scoping

The sandbox.workspace field restricts all file operations to a single directory. NullClaw cannot read or write outside this path, even if the underlying OS permissions would allow it.

"sandbox": {
  "workspace": "/output",
  "allowed_commands": ["echo", "cat", "ls", "mkdir", "cp"]
}

Command Allowlists

NullClaw can only execute commands that are explicitly listed in the allowed_commands array. Attempts to run unlisted commands (including shells like bash or sh) are denied.

echo cat ls mkdir cp bash sh rm curl wget

Audit Logging

Every command execution, file access, and API call made by NullClaw is logged. The SaaS platform additionally tracks instance lifecycle events (created, started, stopped) in the usage_logs database table.

Gateway Pairing

For standalone use, NullClaw supports one-time pairing codes that bind a client to a gateway instance. In Docker deployments, pairing is typically disabled because the compose network provides equivalent isolation.

"gateway": {
  "bind": "0.0.0.0:3000",
  "pairing": false  // network isolation replaces pairing
}

6 Hardening Checklist

Verify each item before deploying to production. Items are grouped by layer and ordered by priority.

Container Layer

All containers run as non-root users
Verify with docker exec <id> whoami
Read-only root filesystem enabled
read_only: true in compose file
no-new-privileges security option set
Prevents privilege escalation via setuid/setgid binaries
Custom seccomp profile applied
Only whitelisted syscalls are allowed
Memory and CPU limits configured
Docker deploy.resources.limits set for every service

Network Layer

Services on isolated bridge networks
No cross-tenant network access possible
Only Traefik exposes host ports
Internal services have no direct port bindings in production
TLS termination at Traefik layer
All external traffic encrypted with HTTPS (Let's Encrypt or custom cert)
Rate limiting enabled on all API routes
Per-IP throttling to prevent brute-force and abuse
CORS restricted to dashboard origin
No wildcard origins allowed

Application Layer

JWT_SECRET set to a strong random value
Not the default dev secret; at least 32 characters of entropy
ENCRYPTION_KEY set and secured
32-byte key for AES-256-GCM API key encryption at rest
Bcrypt cost factor ≥ 12
Higher cost factors increase brute-force resistance
Zod validation on all request bodies
No unvalidated user input reaches business logic
Helmet.js middleware enabled
Secure HTTP headers set on all responses

NullClaw Layer

Sandbox backend configured (Landlock/Bubblewrap)
Set sandbox.backend to "auto" or a specific backend
Workspace scoped to /output only
NullClaw cannot access files outside the designated workspace
Command allowlist is minimal
No shells (bash, sh) or network tools (curl, wget) in the allowlist
API keys injected via environment variables
Keys never appear in config files, images, or logs
Resource limits set in NullClaw config
max_memory_mb and max_disk_mb cap agent resource usage
Gateway pairing disabled (or properly configured)
In Docker, network isolation replaces pairing codes
