Express API

Handles authentication (JWT + bcrypt), instance CRUD, pack management, and container lifecycle. Communicates with Docker via the mounted socket to create, start, stop, and remove NullClaw containers.

React Dashboard

Vite-powered SPA served by nginx. Provides the user interface for registering, creating instances from packs, monitoring status, and viewing logs. Communicates exclusively through the API.

2 Multi-Tenancy

The platform uses a container-per-instance model. Every NullClaw instance runs in its own Docker container with dedicated resources, isolated networking, and a private data volume.

One Container Per Instance

Rather than sharing a single NullClaw process, each user instance gets its own container. This guarantees complete memory isolation, independent crash recovery, and per-instance resource limits.

// API creates a new container for each instance
const containerId = await createContainer({
  instanceId,
  config: JSON.stringify(config),
  port    // random port in 10000-15000 range
})

Network Isolation

Each instance container is placed on the platform-net bridge network. Containers can only reach the API server; they cannot communicate with other instance containers. Port mappings use random high ports to avoid collisions.

Per-Instance Volumes

Every instance gets its own Docker volume for NullClaw's memory database, generated content, and workspace files. Volumes persist across container restarts and are cleaned up when the instance is deleted.

3 Office Packs

Office packs are pre-configured NullClaw templates. Each pack includes a tuned config with a specific AI model, autonomy level, command allowlist, and optional cron schedules. Users select a pack when creating a new instance.

4 Instance Lifecycle

Each NullClaw instance follows a defined lifecycle from creation to deletion. The API manages transitions and records every event in the usage logs.

Start / Stop

Running instances can be stopped to free resources without losing data. Stopped instances retain their volume and config. Starting re-creates the container from the stored configuration.

Error Recovery

If container creation fails, the instance status is set to error. Users can retry by starting the instance again, or delete it and create a new one.

5 API Reference

The Express API exposes RESTful endpoints for authentication, instance management, pack browsing, and log access. All instance routes require a valid JWT token in the Authorization: Bearer <token> header.

Method Endpoint Auth Description
POST /api/auth/register No Create a new user account
POST /api/auth/login No Authenticate and receive JWT
GET /api/instances JWT List all user instances
POST /api/instances JWT Create a new NullClaw instance
POST /api/instances/:id/start JWT Start a stopped instance
POST /api/instances/:id/stop JWT Stop a running instance
DELETE /api/instances/:id JWT Delete instance and clean up
GET /api/packs JWT List available office packs
GET /api/instances/:id/logs JWT Stream instance container logs

6 Scaling

The single-host Docker Compose setup works well for small deployments (tens of instances). As you grow, you will need to consider orchestration platforms and load distribution.

When to Move Beyond Compose

Consider migrating when you encounter any of these signals:

Host memory exceeds 80% with instance containers
More than 50 concurrent instances needed
You need zero-downtime deployments
High availability is a requirement

Docker Swarm

The natural next step from Compose. Swarm mode uses the same compose file syntax with minimal changes. Good for small-to-medium scale (100s of instances).

Familiar compose syntax
Built into Docker Engine
Multi-node overlay networks
Limited scheduling flexibility

Kubernetes

For large-scale production. Kubernetes provides advanced scheduling, auto-scaling, rolling updates, and a rich ecosystem. Best for 1000+ instances or multi-region deployments.

Auto-scaling with HPA
Rolling deploys + canary
NetworkPolicy isolation
Higher operational complexity

Load Balancer Considerations

When scaling beyond a single host, you need an external load balancer in front of Traefik. Key considerations:

1

Sticky sessions for WebSocket

NullClaw gateway connections may use WebSocket for streaming. Configure session affinity (cookie-based or IP-hash) so connections stay on the same node.

2

Health check endpoints

Expose a /healthz route on the API server. Configure the LB to remove unhealthy nodes from the pool.

3

Database migration

SQLite is single-node only. Move to PostgreSQL or MySQL when deploying across multiple nodes. The API uses better-sqlite3 with standard SQL, making migration straightforward.

4

Shared storage

Per-instance Docker volumes are node-local. For multi-node deployments, use a distributed filesystem (NFS, EFS, Ceph) or object storage for shared data.
